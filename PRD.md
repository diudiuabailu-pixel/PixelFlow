> **版本**: v1.2
> **最后更新**: 2026-03-26
> **状态**: MVP 范围收敛中 / 可进入产品设计阶段
> **对标产品**: TapNow (tapnow.ai)

---

## 1. 产品概述

| 项目 | 内容 |
|------|------|
| **产品名称** | PixelFlow |
| **一句话定位** | AI 原生的视觉创作工作台——从脚本到画面，从模板到工作流，一个画布搞定 |
| **核心价值** | 多模型聚合 + 无限画布 + 节点式工作流 + 模板复用 |
| **目标市场** | 中文生态优先，逐步扩展到东南亚 |
| **产品形态** | Web SaaS |
| **商业模式** | Freemium + 按量付费 + 企业订阅 |

### 1.1 我们要解决什么问题

当前 AI 视觉创作的主要问题不是“模型不够多”，而是：

| 问题 | 具体表现 |
|------|---------|
| **工具碎片化** | 生图、生成视频、剪辑、排版分散在多个产品中 |
| **模型分裂** | 每个模型都要单独学习、单独注册、单独付费 |
| **工作流断裂** | 从脚本 → 分镜 → 图片 → 视频的过程上下文不断丢失 |
| **复用困难** | 好的流程难以保存、迁移、复制、共享 |
| **门槛过高** | ComfyUI 很强，但普通创作者很难真正上手 |

### 1.2 我们的答案

> 一个画布，所有模型，完整工作流。

用户不需要在 5~10 个 AI 工具之间反复切换，也不需要先学会复杂节点系统。
PixelFlow 应该让用户在浏览器里完成：

- 输入创意
- 选择模板
- 拖拽工作流
- 生成图片 / 视频
- 保存项目
- 复用流程

### 1.3 产品愿景

**短期（3~6 个月）**：做出一个能跑通“模板 → 画布 → 图像 / 视频生成 → 导出”的 MVP  
**中期（1 年）**：形成模板 / Recipe / 工作流复用生态  
**长期（2 年）**：成为创作者、品牌团队、代理商的 AI 视觉工作台

## 4. 产品核心判断

PixelFlow 的本质不是“另一个 AI 视频生成站”，而是：

> **一个能把内容发现、模板复用、项目管理、模型调用和创作工作流统一起来的视觉创作系统。**

这意味着产品至少要有四层：

1. **内容层**：展示作品、案例、灵感
2. **模板层**：沉淀工作流模板
3. **生产层**：Workspace + Project + Canvas
4. **模型层**：统一调度图片 / 视频模型

如果没有这四层，产品就只是“壳子更厚一点的生成站”。

## 7. 核心对象关系

这是当前版本最容易被忽略、但后续最容易返工的地方。

### 7.1 对象定义

| 对象 | 含义 |
|------|------|
| **Template** | 平台预置或官方整理的可复用工作流模板 |
| **Project** | 用户自己的创作项目，是一个业务容器 |
| **Canvas** | Project 内的可视化工作区 |
| **Node** | Canvas 上的最小创作单元 |
| **Generation** | 一次模型运行任务 |
| **Output Asset** | Generation 产出的图片 / 视频结果 |
| **Recipe / Workflow** | 对工作流结构与参数的抽象表达 |

### 7.2 对象关系

- **Template** 可以被用户 `Clone` 成一个 **Project**
- 一个 **Project** 对应一个 **Canvas**
- 一个 **Canvas** 包含多个 **Node**
- 一个 **Node** 可以触发一到多次 **Generation**
- 一次 **Generation** 会产出一个或多个 **Output Asset**
- **Recipe / Workflow** 可以来源于 Template，也可以从 Project 中沉淀出来

### 7.3 关键设计原则

1. **Template 不是 Project**
   - Template 是可复用蓝本
   - Project 是用户自己的实例

2. **Generation 不是最终资产本身**
   - Generation 是任务记录
   - Output Asset 才是结果文件

3. **Project 必须可反向沉淀为 Workflow / Recipe**
   - 否则平台无法形成复用与社区资产

## 8. MVP 范围（重新收敛版）

这一版最重要的调整，就是把原来过胖的 P0 缩成真正能落地的 MVP。

## 7.1 MVP 核心原则
第一版只验证一件事：

> **用户能否通过模板或基础画布，完成一次从输入到图像/视频输出的创作闭环。**

## 7.2 MVP 必须有

| 功能 | 描述 | 验收标准 |
|------|------|---------|
| Landing Page | 产品介绍与注册入口 | 可完成注册转化 |
| 用户系统 | 登录 / 注册 / 个人基础信息 | 支持邮箱 / Google，微信可后补 |
| Explore 基础作品流 | 展示示例作品 | 可浏览、可点击查看 |
| Templates 基础模板库 | 至少 12~20 个模板 | 分类 + 搜索 + 预览 |
| Workspace | 项目列表页 | 可新建、查看、删除项目 |
| Canvas 基础版 | 无限画布 + 节点 + 连线 | 可编辑基础流程 |
| 基础节点系统 | 输入节点 / 图像生成节点 / 视频生成节点 / 输出节点 | 可运行 |
| 图像生成 | 至少接 2 个图像模型 | 可成功生成 |
| 视频生成 | 至少接 1 个视频模型 | 可成功生成短视频 |
| Output 历史 | 历史结果可查看 / 下载 | 可回溯 |
| Clone Workflow | 模板可复制到 Workspace | 闭环成立 |

## 7.3 MVP 明确不做

第一版先不做：

- 团队协作
- 模板市场
- 社区评论
- 复杂多分支节点
- 视频分析反推
- 白标版
- 企业权限系统
- 高级自动化管线

这部分明确写出来，是为了防止后面自己把自己坑死。

## 8. 节点系统定义（第一阶段）

### 8.1 第一阶段只保留四类节点

#### A. 输入节点
- 文本输入
- 参考图上传

#### B. 生成节点
- 图像生成
- 视频生成

#### C. 处理节点
- 基础参数调节（可并入生成节点）

#### D. 输出节点
- 图片输出
- 视频输出

### 8.3 节点数据模型（示意）

```json
{
  "node": {
    "id": "string (uuid)",
    "type": "input | generate_image | generate_video | process | output",
    "position": { "x": 0, "y": 0 },
    "data": {
      "label": "string",
      "model_id": "string | null",
      "params": {
        "prompt": "string",
        "negative_prompt": "string",
        "width": 1024,
        "height": 1024,
        "steps": 30,
        "seed": -1
      },
      "input_refs": ["node_id:output_key"],
      "output": {
        "type": "image | video | text",
        "url": "string | null",
        "status": "idle | running | success | failed"
      }
    }
  },
  "edge": {
    "id": "string (uuid)",
    "source": "node_id",
    "source_handle": "output_key",
    "target": "node_id",
    "target_handle": "input_key"
  }
}
```

> 此为第一阶段参考结构，具体字段在技术设计阶段可调整。核心原则：结构尽量扁平，方便序列化存储和前端渲染。

### 8.2 为什么要收这么狠
因为第一版最容易死在：
- 节点类型太多
- 数据结构太复杂
- 前端画布交互太重
- 后端任务流太乱

所以先把“能闭环”的最小节点系统做出来，后面再加高级节点。

## 9. 核心用户流程

### 9.1 模板起步路径（推荐第一版主路径）

```text
登录 → 进入 Templates → 选择模板 → Clone 到 Workspace → 打开 Project → 修改 Prompt / 参数 → 运行 → 得到图片/视频 → 下载/保存
```

### 9.2 空白画布路径（进阶用户）

```text
登录 → 新建项目 → 进入 Canvas → 拖入输入节点 → 拖入生成节点 → 选择模型 → 运行 → 输出结果
```

### 9.3 设计原则
第一版不要强迫所有用户从空白画布开始。
**模板路径必须是第一入口。**

### 9.4 异常场景处理

| 异常场景 | 用户感知 | 处理方案 |
|---------|---------|---------|
| **生成失败** | 模型返回错误 | 显示友好提示 + 提供"重试"按钮，不扣额度 |
| **生成超时** | 等待超过 60 秒无响应 | 显示进度状态，允许取消并重试，不扣额度 |
| **模型不可用** | 某模型暂时下线 | 自动推荐替代模型，标注当前模型状态 |
| **额度耗尽** | 用户额度用完 | 生成前预检额度，不足时引导充值/升级，不要等生成完再告知 |
| **文件上传失败** | 参考图上传出错 | 提示格式/大小限制，支持重新上传 |
| **项目保存失败** | 网络中断导致保存异常 | 本地自动缓存 + 恢复网络后自动同步，提示用户 |
| **画布卡顿** | 节点过多导致性能下降 | 设置 MVP 阶段单项目节点上限（建议 50 个），超出时提示 |

## 10. 商业模式

### 10.1 第一阶段商业化方式
- 免费试用额度
- Pro 订阅
- 按量补充额度

### 10.2 收费逻辑
- 不同模型消耗不同额度
- 视频生成消耗高于图片生成
- 高频用户通过订阅获得更低边际成本

### 10.3 定价参考

| 套餐 | 月价（参考） | 包含额度 | 定位 |
|------|------------|---------|------|
| **Free** | ¥0 | 50 积分/月 | 体验产品，约可生成 10 张图或 2 段短视频 |
| **Pro** | ¥49~79/月 | 500 积分/月 | 个人创作者主力套餐 |
| **Pro+** | ¥149~199/月 | 2000 积分/月 | 高频创作者 / 小型电商团队 |
| **按量补充** | ¥1/积分起 | 按需购买 | 突发用量补充 |

> **积分消耗参考**：图像生成 ≈ 5 积分/张，视频生成 ≈ 20~50 积分/段（视模型和时长）

#### 竞品定价对比

| 产品 | 免费额度 | 付费起步价 |
|------|---------|-----------|
| TapNow | 有限免费 | ~$10/月 |

### 10.4 第一阶段先不碰的商业化
- 模板市场分成
- 团队套餐复杂分级
- OEM 白标

|------|---------|------|
| 前端 | Next.js + React + Tailwind + shadcn/ui | 开发快，生态成熟 |
| 画布 | React Flow / Xyflow | 最快落地节点画布 |
| 状态管理 | Zustand | 适合画布交互 |
| 后端 | Next.js API / Node 服务 | 第一版够用 |
| 数据库 | PostgreSQL / Supabase | 省事 |
| 缓存 / 队列 | Redis / Upstash | 生成任务异步调度 |
| 文件存储 | S3 / R2 | 结果文件存储 |

### 11.2 核心技术难点
1. 节点数据结构设计
2. 生成任务调度与状态同步
3. 模型接入层统一抽象
4. 历史结果和项目状态保存

## 12. 产品成功标准

### 12.1 MVP 成功标准
如果第一版要算成功，至少需要满足：

1. 新用户可以在 **3 分钟内** 完成第一次生成
2. 模板 → Workspace → 生成 → 导出的闭环稳定可用
3. 至少有一个图像模型和一个视频模型稳定可用
4. 用户可以保存项目并重新打开继续编辑
5. 历史结果可以回看和下载

### 12.2 MVP 量化验收标准

| 验收项 | 目标 |
|--------|------|
| 注册成功 | 用户能成功注册并进入产品主界面 |
| 模板发现 | 新用户在 1 分钟内能找到可用模板 |
| 模板克隆 | Template → Workspace → Project 路径稳定可用 |
| 首次生成 | 新用户首次从模板生成结果的路径 ≤ 3 分钟 |
| 图像生成 | 至少 2 个图像模型接入成功且稳定返回结果 |
| 视频生成 | 至少 1 个视频模型接入成功且稳定返回结果 |
| 项目保存 | 项目编辑后刷新页面仍可恢复 |
| 历史回看 | 历史生成结果可查看、可下载、可重新生成 |
| 失败反馈 | 任务失败时用户能看到明确错误状态，而不是静默失败 |
| 额度反馈 | 当额度不足时，用户能明确知道原因和下一步操作 |

### 12.3 产品指标

| 指标 | MVP 目标值 | 说明 |
|------|-----------|------|
| 注册转化率 | ≥ 8% | Landing Page 访客 → 完成注册 |
| 首次生成成功率 | ≥ 90% | 新用户第一次点击生成后成功出图/出视频 |
| 首次生成时间 | ≤ 3 分钟 | 从登录到完成第一次生成的端到端时间 |
| 模板使用率 | ≥ 60% | 使用模板起步的用户占比（vs 空白画布） |
| 项目创建率 | ≥ 50% | 注册用户中创建过至少一个项目的比例 |
| 结果下载率 | ≥ 40% | 生成结果被下载的比例 |
| 7 日留存率 | ≥ 20% | 注册 7 天后仍活跃的用户比例 |
| 试用转订阅率 | ≥ 3% | 免费用户 → 付费订阅 |

## 15. 风险评估与应对

| # | 风险 | 影响 | 概率 | 应对策略 |
|---|------|------|------|---------|
| 1 | **范围蔓延** — 开发过程中不断加需求 | 高 | 高 | 严格执行 MVP 不做清单（7.3），需求变更需 PRD 评审 |
| 2 | **模型供应商不稳定** — API 下线、限流、涨价 | 高 | 中 | 接入层做统一抽象，每类能力至少接 2 个供应商，支持快速切换 |
| 3 | **画布性能瓶颈** — 节点多时卡顿 | 中 | 中 | MVP 阶段设节点上限，采用虚拟化渲染，延迟加载非视口节点 |
| 4 | **用户获取困难** — 冷启动无内容无用户 | 高 | 中 | 官方预制 20+ 高质量模板，种子用户邀请制，早期补贴生成额度 |
| 5 | **资金消耗过快** — 模型调用成本高于预期 | 高 | 中 | 设用户日/月生成上限，监控单用户成本，必要时调整免费额度 |

## 16. 最终结论

PixelFlow 方向是对的，但第一版最大的风险不是做不出来，而是：

> **做得太大、太散、太重。**

所以这一版 PRD 的核心调整原则是：

1. **把产品说清楚**
2. **把页面落下来**
3. **把 MVP 收紧**
4. **把不该第一版做的东西明确砍掉**

如果按这版走，PixelFlow 更有机会从一个“很会讲故事的产品概念”，变成一个**真的能上线的第一版产品**。
